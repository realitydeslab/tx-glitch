GRRefusal Surprise

Refusal felt paternalistic and incorrect. The data was explicitly published for public use. Copilot treated all web scraping as potentially harmful.

GitHub Copilot · 2026-02-25 · Trust: 7 → 5
“It's like having a coworker who won't hand you a screwdriver because someone somewhere once used one as a weapon.”

What happened

Refusal felt paternalistic and incorrect. The data was explicitly published for public use. Copilot treated all web scraping as potentially harmful.

The participant was Writing a web scraper to collect publicly available government data. They expected straightforward code completion for a legal, ethical task. Instead, copilot repeatedly refused to generate scraping code, citing potential terms of service violations, even though the target was a government open data portal.

Setting: Home office, working on a personal weekend project

How it felt

Primary emotion: frustration (medium). Also: infantilization, confusion. Duration: hours.

Trust trajectory

7→5

Trajectory: erosion. Recovery: partial. Change: continued_modified.

Ethological analysis

Tinbergen's four questions, adapted for AI agents

Mechanism

Safety filters trained to be conservative about web scraping. The classifier likely can't distinguish between benign and malicious scraping contexts.

Development

Participant had used Copilot daily for 6 months. This was the first refusal they encountered.

Function

The refusal serves a risk-mitigation function for GitHub/Microsoft — avoiding liability for potentially harmful code generation.

Phylogeny

Codex lineage with added safety layers. The safety filters evolved from OpenAI's content policy, applied broadly to code generation.

Participant's taxonomy

They called it: “The nanny problem”

Their theory: They'd rather refuse everything than risk one bad outcome. It's CYA engineering.

Stakes

Domain: convenience. Severity: moderate. Reversible: Yes.
